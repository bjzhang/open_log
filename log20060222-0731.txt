c语言***
自：http://www.ednchina.com/BBS/ShowTopic.aspx?id=6267。想成为嵌入式程序员应知道的0x10个基本问题。
12 . 下面的代码输出是什么，为什么？
　　void foo(void)
　　{
　　unsigned int a = 6;
　　int b = -20;
　　(a+b > 6) ? puts("> 6") : puts("<= 6");
　　}
　　这个问题测试你是否懂得C语言中的整数自动转换原则，我发现有些开发者懂得极少这些东西。不管如何，这无符号整型问题的答案是输出是 ">6"。原因是当表达式中存在有符号类型和无符号类型时所有的操作数都自动转换为无符号类型。因此-20变成了一个非常大的正整数，所以该表达式计算出的结果大于6。这一点对于应当频繁用到无符号数据类型的嵌入式系统来说是丰常重要的。如果你答错了这个问题，你也就到了得不到这份工作的边缘。

编程，用法，c语言，verilog
对关系运算符的误用：如果i=6，j=3<i<5?i:0;的结果是什么？
答案：j=6，因为关系运算符是左结合的（与加减一样）。所以是先计算3<i，结果是真(c语言中是1)，再计算1<6，结果还是真，所以j=6。关系运算符的这种用法符合语法所以不会报错，但是可能得不到预期的逻辑，编写代码时一定要注意！！！

关于skyeye1.0版本
自：http://www.linuxfans.org/nuke/modules.php?name=Forums&file=viewtopic&t=149525
采用gdb远程调试的理由如下：代码结构清晰，容易扩展，因为以前和gdb的代码混合在一起，乱乱的，很难支持多个架构和加一些自己的扩展。另外采用远程调试的功能，也容易和IDE集成. 关于这个问题的论述，我会添加到UserManual中 
关于调试的功能，目前正在完善，确实还有些bug,不稳定。所以如果需要调试功能的话，我建议还是使用1.0以前的版本 
 
实际上，我也比较喜欢用1.0以前版本的SkyEye做一些调试，确实要比现在开两个窗口调试方便一些。但是从模块化考虑或者代码维护的角度考虑，和gdb混杂在一起，也给我们带来了很多不方便，而且我们以后会加一些性能测试分析，或者多处理器，多架构的一些东西进去，和gdb分开的话有利于以后的扩展。这是个比较困难的选择，但是我们还是认为应该和gdb分开。从理论上讲，这样整体架构很清晰，我们完全把SkyEye当作一块开发板，可以用各种各样的调试手段去调试，jtag,gdb rsp，ads等等，完全是一种C/S的模式。 
另外一个考虑就是，也可以在SkyEye的模拟模块中做一个gdb接口文件，可以以库的形式提供给gdb,让gdb编译的时候，把SkyEye编译进去来兼容SkyEye 1.0以前的用法。希望我有时间能够完成它。。。 
目前，大家需要调试的话，我建议大家用1.0以前的版本，1.0更高的版本中的gdb 调试功能应该在下一个release中，完善。。。

在linux中设置路径：
1,如果修改整个系统的路径 vi /etc/profile (有的文章说是.profile, 可能是版本不同), 每个用户的配置文件文件是 $HOME/.bash_profile, $HOME/.bashrc, 设置路径一般在$HOME/.bash_profile (是隐藏文件,用ls -a查看)
如果定义了pathmunge函数直接在unset之前加入
pathmunge 路径 [after]
after表示加入到当前路径的最后, 否则加入到当前路径的起始处
例如:
pathmunge /opt/sparc-linux/bin/ after
在当前路径后加入/opt/sparc-linux/bin/ (sparc 开发工具链的路径)
完成后,":wq",存盘退出.
使用source /etc/profile可以使刚才的修改立即生效.
如果没有pathmunge函数, 用下面的语句(不要加空格):
PATH="$PATH":路径
2,如果只是临时修改, 直接在shell中输入下面的命令
PATH="$PATH":路径
修改完成后, 可以用
env $PATH
查看路径是否设置正确

添加应用程序:
方法1,查看由make xconfig ( or menuconfig, config等)生成的Makefile的开始部分, config ... 有一句
all: subdirs romfs modules modules_install image
表明, 如果直接make, 等价于make subdirs, make romfs, make modules, make modules_install, make image.
容易看出make romfs是生成romfs目录的命令(经过验证,的确如此). 所以有一个简单的添加应用程序的方法:
手动进行make all规则的所有步骤. 在make romfs之后, 把自己已经编译好的应用程序复制到romfs目录. 然后继续make all的其它规则.
这样在生成的image.*中就包括了自己编译的应用程序.这样加入应用程序完全需要自己编译,设定编译选项.需要理解uClinux编译选项中所有与应用程序有关的内容.例如"CC"和"LDFLAG".

    如今的时钟信号正变得越来越快，cpu的基频都已经达到甚至超过了200Mhz, 时钟偏移，抖动，以及传输延时对时钟系统的影响也越来越大，在设计中就要充分考虑到时终裕度，也就是要提前进行一些时序的预算，以确保在最坏的情况下系统都能稳定的工作。
    在当今常见的同步设计中，驱动各种元件通常需要多个时钟信号。要复制所需数量的时钟，需要建立时钟树。时钟树从一个时钟脉冲源（如振荡器或外部信号）开始，驱动一个或多个缓冲器。缓冲器的数量通常取决于目标器件的数量和位置。时钟驱动器有两种主要类型：缓冲器类型的器件（不带锁相环）和反馈类型的器件（带锁相环）。后者现在比较流行的都是zdb（零延时缓冲器），它是一种可以将一个时钟信号扇出成多个时钟信号，并使这些输出之间有零延时和很低的偏斜的器件。此器件很适合用于要求输入到输出和输出到输入的偏斜极小的各种时钟分发应用中。很多ZDB都有一个外部反馈通路开路，可以很方便的用来进行一些提前或滞后的调节。当然，还有一些比较专用的时钟缓冲器，用在要求非常精确的时钟子系统中，比如SDRAM时钟缓冲器。差分时钟，可以有效抑制COMMON NOISE,应用也是越来越广泛，在近几年，差分PECL、LVPECL和LVDS时钟成为对高速逻辑电路定时的非常流行的方法 。
     还有时钟一般都在相当高的频率下工作，而且负载相当重，因为要连接到每个触发器，功率的消耗也是相当的可观，有必要选择本身消耗功率比较低的时钟缓冲器和发生器，或进行必要的散热措施。在时钟电路中，layout也是相当的重要，一些guideline必须要遵循，芯片厂商一般都有提供。
     在设计中，曾用到过cypress, realtek的时钟发生器和缓冲器，CYPRESS在这方面相当知名，REALTEK的也不错，价格相对来说也比较便宜。

     时钟电路的设计相当复杂，还在不断地摸索和学习中。欢迎大家踊跃交流，共同提高。

上网时间 : 2006年06月29日 
  
 打 印 版    推 荐 给 同 仁    发 送 查 询 
 
三星电子在近日推出了首款闪存笔记本电脑。这款笔记本以32GB的NAND闪存完全替代了传统的硬盘驱动器。 

早在去年9月，三星与希捷在闪存是否会全面取代硬盘技术上展开争论，三星高层预言，硬盘时代已步入黄昏，闪存时代即将来临。希捷公司随即反驳，认为闪存完全替代硬盘还为时尚早。 

易观国际认为，从目前笔记本发展的趋势来看，移动性强，省电等指标已经成为用户购买笔记本电脑时的重要考虑因素，未来笔记本将进一步在不影响性能的前提下向轻薄化和长电池使用时间发展，闪存技术凭借其耗电量低和体积小的特点可以有效地满足这样的需求。但闪存技术由于价格昂贵限制了其大范围的使用，随着技术的发展和产量的提升，闪存的价格将进一步下降，预计未来一到两年将有少量基于闪存的高端轻薄笔记本在市场销售，随后将逐渐在笔记本领域普及。 

易观国际还指出，闪存技术无论在性能还是使用方便性上都优于硬盘，但其高昂的成本使其目前在性价比上远落后于硬盘，随着用户需求的提升和闪存价格的下降，闪存技术将在5-8年内会在笔记本上广泛使用，但闪存从根本上取代硬盘是不可能的，在服务器工作站等领域，硬盘凭借其存储量大及价格便宜占有更好的优势，未来硬盘的发展将主要集中在这些领域，而在笔记本上将采用闪存技术。

主要看你开发什么类型产品的。总体来说SIRF芯片方案的模块比较多。毕竟人家做的早嘛。不过各家性能也不一样的，功耗大。价格稍贵。另外我建议nemerix的芯片的。性价比很高。其中有一个可以天线内外自动切换，这个功能我觉得比较好。
snsun2008@hotmail.com 
RF:
现在GPS芯片的解决方案已经很成熟了，如主流的SIRF，ATMEL等厂商都可以提供单芯片的解决方案，其BASEBAND部分只需很少的外部电路就可以完成设计。但对于RFBAND部分来说，可能有点麻烦了，因为民用GPS的信号载频是1575.42MHz，这就牵涉到RF电路设计了。
RF电路相较一般的数字电路（高速数字电路与RF设计有些类似了）而言,要显得麻烦很多。而且RF电路调试一般都需要比较贵重的仪器，如网络分析仪，噪声仪等。这对于想从事GPS硬件设计的个人朋友或小型公司来说，着实是一道过不了的坎。如果再加上什么LNA的设计，前端接收天线的匹配等，要是没有相关仪器及RF设计经验，要想在短时间内设计成功，可以说很难。
   总之，GPS模块的设计，RF部分设计是绕不过的关口。有人讲RF设计是一门“黑盒子”艺术，在设计调试时很容易出现一些意想不到的莫明其妙的问题。当然这些问题并不是难以解决的，不过解决问题所需要的资源和时间，可就与设计者所具有的相关理论知识及调试设计经验紧密相关了。
    有问题欢迎共同探讨。
